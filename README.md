# Автоматизированное тестирование API Swagger PetStore

## Описание проекта

Проект автоматизации тестирования REST API для [Swagger PetStore](https://petstore.swagger.io/)

#### Фокус проекта:

- Построение масштабируемой и поддерживаемой архитектуры автотестов
- Покрытие API позитивными и негативными сценариями
- Проверка контрактов и бизнес-логики
- Выявление дефектов публичного API

#### Проект демонстрирует: 

- Проектирование архитектуры тестового фреймворка
- Разделение ответственности 
- Написание позитивных и негативных тестов
- Применение параметризации в тестах
- Валидация JSON Schema
- Генерация тестовых данных
- Настройка логирования и отчетности 

## Архитектура проекта

#### Конфигурация

- Централизованная настройка
- Request/Response спецификации
- Логирование

#### Модели данных

- POJO классы
- Jackson для сериализации и десериализации JSON

#### Клиенты

- Классы-клиенты для работы с эндпойнтами
- Поддержка raw-запросов для негативных сценариев

#### Данные

- Генерация тестовых данных (Data Faker)
- Отдельные наборы данных для негативных сценариев

#### Тесты

- Базовый тестовый класс с общими настройками
- Разделение тестов по эндпойнтам

## Технологии

- Язык: Java 17
- Тестовый фреймворк: JUnit 5
- Работа с API: RestAssured
- Сборка: Maven
- Отчетность: Allure
- Данные:
  - Data Faker (генерация),
  - Jackson (сериализация/десериализация JSON)


## Тестовые сценарии

Покрытие эндпойнтов /pet:

- Позитивные CRUD тесты (создание, обновление, получение, удаление)
- Поиск по статусу (с применением параметризации)
- Валидация JSON Schema
- Негативные тесты
  - отсутствующее или некорректное тело запроса
  - отсутствующий или невалидный id
  - отсутствующие обязательные поля
  - дублирование id
  - несовпадение ContentType
  - несовпадение типов данных

## Примеры тестов

Ниже приведены примеры позитивного и негативного сценариев:

### Позитивный сценарий: Создание питомца с полным набором данных

     @Test
     public void createPet_withFullData() {

        // ARRANGE
        Pet pet = PetFactory.randomPetFullData();

        // ACT
        Response response = petClient.createPet(pet);

        // ASSERT: Проверка ответа
        response
                .then()
                .spec(okResponse)
                .body("id", greaterThan(0L));

        // ASSERT: Проверка содержания ответа
        Pet createdPet = extractPet(response); // централизованная десериализация ответа API в объект Pet
        pet.setId(createdPet.getId());
        assertPetFieldsEquals(pet, createdPet); // сравнение всех полей объектов

        // ASSERT: Проверка состояния данных
        Pet fetchedPet = extractPet(petClient.getPetById(createdPet.getId()));
        assertPetFieldsEquals(pet, fetchedPet);
    }

Тест демонстрирует полный цикл проверки API: 
создание сущности, валидацию ответа и проверку сохранения данных в системе.

### Негативный сценарий: Создание питомца с уже существующим id

    @Test
    public void createPet_withDuplicateId() {

        // ARRANGE
        Pet pet = PetFactory.randomPetFullData();
        Long id = extractPet(petClient.createPet(pet)).getId();

        Pet duplicatePet = PetFactory.randomPetFullData()
                .setId(id);

        // ACT
        Response response = petClient.createPet(duplicatePet);

        // ASSERT
        response
                .then()
                .spec(duplicateIdResponse);
    }

Результат выполнения теста:

    ОР: 409 Conflict (возможен 400 Bad Request)
    ФР: 200 OK, существующая сущность перезаписана

## Результаты тестирования

Все позитивные сценарии выполняются успешно.

Часть негативных тестов выявляет дефекты API, описанные ниже.

### Найденные баги

*Примечание: Указанные `id` в примерах данных (в т.ч. в ФР) условные. В реальных тестах используются `id`, сгенерированные API*

#### POST /pet

1. Создание сущности без обязательных полей

| Сценарий                  | Пример данных                | Ожидаемый результат                               | Фактический результат                 |
| ------------------------- |------------------------------|---------------------------------------------------|---------------------------------------|
| 1.1. Без всех полей       | `{}`                         | 400 Bad Request (возможен 405 Method Not Allowed) | 200 OK, сущность создаётся c `id`:153 |
| 1.2. Без поля `name`      | `{"photoUrls": ["some url"]}`| 400 Bad Request (возможен 405 Method Not Allowed) | 200 OK, сущность создаётся c `id`:156 |
| 1.3. Без поля `photoUrls` | `{"name": "Kitty"}`          | 400 Bad Request (возможен 405 Method Not Allowed) | 200 OK, сущность создаётся c `id`:158 |
    

2. Перезапись сущности при дублировании id

| Предусловие                          | Пример данных                                             | Ожидаемый результат                     | Фактический результат                      |
|--------------------------------------|-----------------------------------------------------------| --------------------------------------- | ------------------------------------------ |
| В системе создан питомец с `id`: 101 | `{"id": 101, "name": "Kitty", "photoUrls": ["some url"]}` | 409 Conflict (возможен 400 Bad Request)                 | 200 OK, существующая сущность перезаписана |

3. Ошибка сервера при невалидной структуре JSON

| Пример данных                                | Ожидаемый результат                               | Фактический результат     |
|----------------------------------------------|---------------------------------------------------| ------------------------- |
| `{"name": "Bobik", "photoUrls": "notArray"}` | 400 Bad Request (возможен 405 Method Not Allowed) | 500 Internal Server Error |
